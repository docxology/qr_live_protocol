# QR Live Protocol Remediation ISA

## Problem

QR Live Protocol has useful local QR generation, signing, HMAC, encryption, and web-display pieces, but the implementation and documentation overstate what a third-party viewer can verify. The current verifier is mostly same-process or same-secret, the web `/api/verify` endpoint only validates JSON shape, and several tests fail against the advertised behavior.

## Vision

QRLP should be honest and technically defensible: a local streaming overlay may use private HMACs for operator-side integrity, while public authenticity depends on a signed payload and an explicitly trusted issuer public key. The default developer experience should install, test, generate, and verify without hidden local state surprises.

## Out of Scope

This remediation does not turn QRLP into a decentralized identity network, certificate authority, on-chain commitment protocol, native mobile scanner app, or hardened internet-facing service. Blockchain hashes remain freshness context unless a future version commits QR/content hashes to a chain.

## Principles

- Public verification must be based on public verifier material, not copied private keys or process-local HMAC secrets.
- HMAC is a private integrity primitive; signatures are the public authenticity primitive.
- Verification endpoints must fail closed for malformed, unsigned, tampered, expired, or untrusted payloads.
- Documentation must describe implemented behavior, not aspirational production architecture.
- Tests should exercise the same boundaries users cross: fresh processes, clean key directories, web API calls, and CLI commands.

## Constraints

- Use the repository's existing Python architecture and cryptography library stack.
- Keep backwards-compatible JSON field names where practical.
- Do not require network blockchain or NTP availability for the default test suite.
- Do not add heavyweight services or new deployment substrates.
- Keep Forge/error-recovery edits disjoint from protocol/authentication edits.

## Goal

Implement a defensible public-verification path for QRLP: generated signed QR payloads carry stable issuer/key metadata, verifiers can validate them with trusted public keys without shared private state, web and CLI verification call the real verifier, packaging resolves in a clean environment, and tests/docs align with the reduced but real security claim.

## Decisions

- 2026-07-09: Treat HMAC as a local integrity feature, not a public verifier feature.
- 2026-07-09: Use explicit trusted public keys instead of trying to infer trust from local identity hashes.
- 2026-07-09: Keep blockchain hashes as context in this pass; durable chain anchoring is out of scope.
- 2026-07-09: Default to failing closed for authenticity while preserving detailed sub-results for diagnostics.

## Changelog

- conjectured: The original same-instance verifier could be repaired by sharing key managers in tests.
  refuted by: Cross-instance probes showed public verification still failed without private/local state.
  learned: The missing abstraction is not a key-manager fixture but an explicit public trust store.
  criterion now: ISC-3 and Anti-21 require verification with only trusted public key material.
